# PRD-005 — Turn Lifecycle

**Project:** Markdown Game Runtime (MGR)

**Version:** 1.1

**Status:** Draft

---

# 1. Mục tiêu

Turn Lifecycle định nghĩa quy trình xử lý một lượt chơi trong MGR.

Mọi Game Package phải sử dụng cùng một Turn Lifecycle.

Gameplay có thể khác nhau.

Lifecycle không được khác.

---

# 2. Triết lý

Một lượt chơi là một Transaction.

Transaction phải:

* có điểm bắt đầu
* có điểm kết thúc
* có thứ tự xử lý rõ ràng
* không được bỏ dở

Mỗi lượt phải hoàn toàn độc lập.

---

# 3. Kiến trúc

```text
Turn N

↓

Input Phase

↓

Validation Phase

↓

Pre Event Phase

↓

Simulation Phase

↓

Post Event Phase

↓

State Commit

↓

Response Phase

↓

Turn N+1
```

Event được chia thành hai pha rõ ràng.

Pre Event thiết lập môi trường trước khi Simulation chạy.

Post Event phản ứng với kết quả của Simulation.

Xem §7 và §7a.

---

# 4. Input Phase

Runtime nhận Input.

Nguồn Input:

* Player
* System
* Internal Event

Input phải được chuẩn hóa trước khi xử lý.

Ví dụ:

```text
Build Factory
```

↓

Action Object

---

# 5. Validation Phase

Validation kiểm tra:

* Action tồn tại
* Điều kiện đủ
* Resource đủ
* Rule cho phép
* State hợp lệ

Nếu thất bại:

Lifecycle kết thúc.

State không thay đổi.

---

# 6. Simulation Phase

Runtime mô phỏng kết quả của Action.

Ví dụ:

```text
Hire Employee
```

↓

* tiêu tiền
* tăng nhân viên
* thay đổi chi phí

Simulation không được ghi State ngay.

Chỉ tính toán.

---

# 7. Pre Event Phase

Trước Simulation.

Runtime kích hoạt các Event thiết lập môi trường của Turn.

Ví dụ:

* Environment Event (thời tiết, thời gian trong ngày)
* Scheduled Event (đã hẹn từ Turn trước)
* Setup Event (biến hidden của Turn được roll)

Pre Event có thể ghi Hidden State và Derived State.

Pre Event KHÔNG được Commit — chỉ chuẩn bị dữ liệu cho Simulation.

# 7a. Post Event Phase

Sau Simulation.

Runtime kích hoạt các Event phản ứng với kết quả của Simulation.

Ví dụ:

* Random Event (xảy ra sau khi Action đã tính)
* Story Event
* NPC Event
* Consequence Event (event phụ thuộc kết quả)

Post Event có thể sinh Action mới cho Turn tiếp theo.

Post Event không được phá Rule.

Post Event chỉ chạy nếu Simulation đã thành công.

---

# 8. Rule Resolution

Nếu nhiều Rule cùng tác động:

Runtime phải giải quyết theo thứ tự ưu tiên.

Ví dụ:

```text
System Rule

↓

Runtime Rule

↓

Game Rule

↓

Event Rule
```

Không được xử lý ngẫu nhiên.

---

# 9. State Commit

Sau khi toàn bộ Simulation hoàn tất.

Runtime mới ghi State.

Không được ghi từng phần.

State Commit phải là thao tác nguyên tử (Atomic).

Nếu xảy ra lỗi trước Commit:

Toàn bộ thay đổi bị hủy.

---

# 10. Response Phase

Sau Commit.

Runtime sinh phản hồi.

Bao gồm:

* Narrative
* Effect Summary
* Updated UI
* Available Actions

Không được sửa State trong bước này.

---

# 11. Turn Complete

Sau khi Response hoàn thành.

Runtime chuyển sang trạng thái:

```text
WAIT PLAYER INPUT
```

Không xử lý gì thêm.

---

# 12. Atomicity

Một Turn phải:

* hoặc thành công hoàn toàn
* hoặc không thay đổi gì

Không tồn tại trạng thái trung gian.

Ví dụ:

Sai:

```text
Money giảm

Employee chưa tăng
```

Đúng:

```text
Money giảm

Employee tăng
```

đồng thời.

---

# 13. Idempotency

Nếu cùng một Turn được xử lý lại do lỗi hệ thống.

Kết quả phải giống nhau.

Không được cộng tài nguyên hai lần.

Không được kích hoạt Event hai lần.

---

# 14. Side Effects

Mọi thay đổi State đều là Side Effect.

Side Effect chỉ được phép sinh ra trong:

* Simulation
* Event

Response không được tạo Side Effect.

UI không được tạo Side Effect.

Narrative không được tạo Side Effect.

---

# 15. Event Queue

Runtime phải xử lý Event theo hàng đợi.

```text
Action

↓

Pre Event Queue

↓

Process Pre Event 1

↓

Process Pre Event 2

↓

Simulation

↓

Post Event Queue

↓

Process Post Event 1

↓

Process Post Event 2

↓

Commit
```

Mỗi pha có hàng đợi riêng.

Không được xử lý Event song song.

Không được đưa Pre Event vào Post Event Queue hoặc ngược lại.

---

# 16. Nested Events

Event có thể sinh Event khác.

Ví dụ:

```text
Festival

↓

Tourist

↓

Economic Boom

↓

Stock Market
```

Runtime phải xử lý tuần tự.

Không được đệ quy vô hạn.

Builder hoặc Runtime phải có giới hạn số tầng Event.

---

# 17. Failure Recovery

Nếu Action thất bại:

Runtime:

* giải thích ngắn gọn
* không đổi State
* không kích hoạt Event
* chờ Input mới

---

# 18. Narrative Timing

Narrative luôn xuất hiện sau khi State đã Commit.

Không được kể trước khi kết quả được xác nhận.

Ví dụ:

Sai:

```text
Bạn thuê được nhân viên.

...

Không đủ tiền.
```

Đúng:

```text
Không đủ tiền.

Việc tuyển dụng thất bại.
```

---

# 19. Turn History

Sau mỗi Turn.

Runtime ghi:

* Turn Number
* Player Action
* Events
* State Changes

History là bất biến.

Không được chỉnh sửa Turn cũ.

---

# 20. Lifecycle Invariants

Mọi Turn phải đảm bảo:

* Input chỉ đọc một lần.
* Validation luôn trước Pre Event.
* Pre Event luôn trước Simulation.
* Simulation luôn trước Post Event.
* Post Event luôn trước Commit.
* Commit luôn trước Response.
* Response không được sửa State.
* State chỉ Commit một lần.
* Không có Event sau Commit.
* Pre Event và Post Event dùng hàng đợi riêng.

---

# 21. Non-goals

Turn Lifecycle không định nghĩa:

* State Schema
* UI
* Save Format
* Plugin
* Random Formula
* Combat Logic
* Economy Logic

Các nội dung này thuộc các PRD khác.

---

# 22. Tiêu chí hoàn thành

Turn Lifecycle được coi là hoàn thành khi:

* Mọi game MGR đều xử lý lượt theo cùng một quy trình.
* Không có thay đổi State ngoài State Commit.
* Không có Narrative đi trước kết quả mô phỏng.
* Mỗi lượt là một Transaction hoàn chỉnh, nguyên tử và có thể kiểm chứng.
* Pre Event và Post Event tách biệt, đủ để mô tả cả setup môi trường và
  hậu quả của Action.
