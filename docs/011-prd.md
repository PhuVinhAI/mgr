# PRD-011 — Rule Execution Model

**Project:** Markdown Game Runtime (MGR)

**Version:** 1.0

**Status:** Draft

---

# 1. Mục tiêu

Rule Execution Model định nghĩa cách Runtime **thực thi** Rule.

PRD-010 đã nói Rule được **viết** như thế nào.

PRD-010 chưa nói:

* thứ tự Effect chạy trong một Rule
* nhiều Rule cùng sửa một biến thì ai thắng
* Rule có atomic không
* rollback khi vi phạm invariant
* read-after-write

Rule Execution Model lấp phần đó.

Nếu không có PRD này, cùng State + cùng Rule + cùng input có thể ra
kết quả khác nhau giữa hai LLM — vi phạm PRD-004 §4 Determinism.

---

# 2. Triết lý

Runtime thực thi Rule như một transaction.

Rule phải:

* deterministic
* snapshot-based (đọc từ Turn Start Snapshot)
* atomic per Rule (một Rule chạy trọn vẹn hoặc không chạy)
* order-explicit (Priority quyết, không dựa vào thứ tự file)
* rollback-safe (vi phạm invariant → bỏ Rule đó, giữ Rule khác)

Rule không được:

* đọc giá trị đã bị mutate trong cùng Effect block
* mutate State ngoài Phase được phép
* gọi Rule khác
* recurse

---

# 3. Kiến trúc

Trong mỗi Phase (Validation / Pre Event / Simulation / Post Event),
Runtime chạy pipeline sáu bước.

```text
Match     — Rule nào có Trigger fire ở Phase này
Sort      — sort theo Priority giảm dần, đồng Priority theo Package order
Prepare   — evaluate Precondition trên Turn Start Snapshot
Apply     — build Pending Delta từ Effect (top-down)
Verify    — kiểm tra invariant sau khi merge Delta vào snapshot ảo
Commit    — merge Delta vào State (chỉ tại State Commit phase)
```

Pending Delta chỉ tồn tại giữa Apply và Commit. Không thấy được từ
Formula hoặc Rule khác đang chạy trong cùng Phase.

---

# 4. Turn Start Snapshot

Turn Start Snapshot là State ở đầu Turn.

Mọi Rule trong Turn đọc từ Turn Start Snapshot cho vế phải của Effect.

Ví dụ:

```text
Rule ApplyBuyLemons

Effect:
Money -= Quantity × 5
Lemons += Quantity
```

`Money` và `Quantity` đọc từ Turn Start Snapshot, không phải từ giá trị
đã bị Rule khác trong cùng Turn cập nhật.

Điều này giữ semantics giống nhau bất kể thứ tự Rule chạy.

---

# 5. Rule Matching

Một Rule được **matched** ở Phase P khi:

* `Kind` phù hợp với Phase P (Guard→Validation, Transformation→Simulation,
  Trigger→Pre Event | Post Event | State Commit)
* `Trigger` fire ở Phase P
* `Precondition` trả `true` khi evaluate trên Turn Start Snapshot

Rule không match thì bị skip. Skip được ghi vào Turn History (PRD-005 §19).

---

# 6. Priority Resolution

Trong danh sách Rule đã match, sắp xếp theo:

1. `Priority` giảm dần (giá trị lớn chạy trước)
2. Cùng `Priority` → theo thứ tự khai báo trong Game Package
3. Cùng `Priority` và cùng file → theo thứ tự dòng

Không được resolve ngẫu nhiên.

`Priority` mặc định = 0.

---

# 7. Effect Evaluation Order

Trong một Rule, Effect block liệt kê từng mutation trên một dòng.

Runtime evaluate từ trên xuống dưới.

Mỗi mutation:

1. Đánh giá vế phải trên Turn Start Snapshot.
2. Ghi cặp `(Rule, Variable, Op, Value)` vào Pending Delta.
3. Không cập nhật snapshot ngay.

Dòng sau **không thấy** dòng trước.

---

# 8. Cấm read-after-write

Trong cùng một Effect block, không được đọc biến vừa bị mutation.

Sai:

```text
Effect:
Money += Bonus
Reputation += Money × 0.01
```

Ở đây `Money × 0.01` đọc Money **sau khi** đã cộng Bonus — không rõ
LLM lấy giá trị nào.

Đúng:

```text
Formula NewMoney
Money + Bonus

Effect:
Money := NewMoney
Reputation += NewMoney × 0.01
```

Named Formula (PRD-009 §14) chạy trên Turn Start Snapshot ⇒
deterministic.

Builder phải phát hiện read-after-write và emit warning
`RULE_EXECUTION_READ_AFTER_WRITE`.

---

# 9. Atomicity per Rule

Một Rule là một đơn vị.

Nếu bất kỳ mutation nào trong Rule fail (type mismatch, không tìm thấy
Reference, kết quả out-of-domain), **toàn bộ Rule** bị drop khỏi Pending
Delta. Các Rule khác trong Phase không bị ảnh hưởng.

Phải all-or-none. Không được ghi một nửa.

---

# 10. Conflict Policy

Nhiều Rule cùng match và cùng ghi vào một Variable trong cùng Phase.

Cách giải quyết phụ thuộc `Op`.

## `:=` (assignment)

Assignment không commutative.

Chỉ được có **một** `:=` với cùng Variable trong cùng Phase.

Nếu có nhiều, Builder emit `RULE_EXECUTION_CONFLICT_ASSIGN` và Build
fail.

Phải phân giải bằng Priority rõ ràng hoặc tách sang Phase khác.

## `+=`, `-=` (additive)

Additive commutative.

Delta cộng dồn theo thứ tự Priority.

```text
Rule A: Money += 10   Priority: 5
Rule B: Money += 20   Priority: 3

Turn Start: Money = 100
Sau Commit: Money = 130
```

## `×=`, `÷=` (multiplicative)

Multiplicative commutative on Rational.

Nhân dồn theo thứ tự Priority.

```text
Rule A: Price ×= 1.10   Priority: 10   (base markup)
Rule B: Price ×= 0.90   Priority: 5    (discount)

Turn Start: Price = 100
Sau Commit: Price = 100 × 1.10 × 0.90 = 99
```

Rounding: chỉ ở lần cuối cùng, theo `Round`/`Floor`/`Ceil` (PRD-009 §17).

## Trộn `:=` và `+=`

Nếu một Variable có cả `:=` và `+=` trong cùng Phase từ các Rule khác
nhau: Builder emit `RULE_EXECUTION_CONFLICT_MIX` và Build fail.

Author phải rewrite bằng Named Formula.

---

# 11. Modifier Layering

Pattern phổ biến: base value + nhiều modifier.

```text
Formula BasePrice
10

Rule SalesModifier
Kind: Trigger
Trigger: On Pre Event
Effect: Price ×= (1 - SaleDiscount)
Priority: 5

Rule DifficultyModifier
Kind: Trigger
Trigger: On Pre Event
Effect: Price ×= DifficultyFactor
Priority: 10

Rule ResetPrice
Kind: Trigger
Trigger: On Pre Event
Effect: Price := BasePrice
Priority: 100
```

Thứ tự áp dụng: `ResetPrice` (Priority 100) đặt Price = BasePrice, rồi
`DifficultyModifier` (10) nhân, rồi `SalesModifier` (5) nhân.

Đây là composition deterministic.

---

# 12. Cross-Phase Execution

`Kind` quyết định Phase.

```text
Guard          → Validation Phase
Transformation → Simulation Phase
Trigger        → Pre Event | Post Event | State Commit
```

Trong mỗi Phase, pipeline §3 chạy độc lập.

Pending Delta của Phase P không carry sang Phase Q. Mỗi Phase commit
Delta của mình vào State ảo cho Phase kế tiếp đọc.

Runtime không được inject Rule của Phase khác. Một Rule chỉ có một
`Kind`.

---

# 13. State Commit

State Commit là điểm duy nhất Delta được ghi vào State thực (PRD-005 §12).

Trước Commit:

1. Merge tất cả Pending Delta từ mọi Phase trong Turn.
2. Kiểm tra State Invariant (PRD-006 §14).
3. Nếu vi phạm: rollback theo §14.

Sau Commit, snapshot mới là Turn Start Snapshot cho Turn kế tiếp.

---

# 14. Rollback

Nếu một Rule khiến State vi phạm invariant (`Money >= 0` sau `Money -= 500`
với `Money = 100`), Runtime thực hiện theo thứ tự sau:

1. Drop **contribution của Rule đó** khỏi Pending Delta (không drop cả
   Delta).
2. Re-verify invariant.
3. Nếu vẫn vi phạm, drop Rule Priority thấp hơn kế tiếp.
4. Lặp cho đến khi hết vi phạm hoặc Pending Delta rỗng.
5. Nếu Delta rỗng và invariant vẫn vi phạm → State đã invalid từ trước
   Turn → emit `STATE_INVARIANT_UNRECOVERABLE` và stop game.

Ghi tất cả các bước vào Turn History.

Không bao giờ commit một phần Rule.

---

# 15. Isolation between Actions

Trong một Turn có thể có nhiều Action (Player Action + Event Actions).

Mỗi Action tạo Pending Delta riêng, chạy hết pipeline §3, rồi merge
vào Turn-level Pending Delta.

Rule chạy cho Action X không thấy Delta của Action Y.

Điều này ngăn Rule vô tình đọc mutation của Action khác.

---

# 16. Effect Dependency Detection

Builder phân tích tĩnh mối phụ thuộc giữa Rules cùng Phase:

* Rule A ghi Variable X.
* Rule B đọc X trong Precondition hoặc Effect.

Nếu cả A và B cùng Phase và cùng `Trigger`, Builder emit warning
`RULE_EXECUTION_ORDER_AMBIGUOUS`.

Author giải quyết bằng cách:

* tách sang hai Phase (A vào Pre Event, B vào Simulation)
* đặt Priority rõ ràng (A cao hơn B nếu muốn A chạy trước — nhưng nhớ
  §4: B vẫn đọc snapshot, không đọc mutation của A trong cùng Turn).

---

# 17. Debug

Turn History (PRD-005 §19) phải ghi cho mỗi Rule:

* Rule name.
* Phase đã chạy.
* Precondition kết quả (true/false).
* Applied | SkippedByPrecondition | DroppedByAtomicity | DroppedByRollback.
* Pending Delta contribution (Variable, Op, Value).

Cho phép replay và audit theo PRD-004 §4.

---

# 18. Interaction với Formula

Formula (PRD-009) là biểu thức thuần túy, không side effect (PRD-009 §16).

Trong Effect Apply, mọi Reference trong Formula đọc từ Turn Start
Snapshot của Phase hiện tại.

Randomness (PRD-009 §13) seed bằng `(Turn Number, Phase, Rule name)`
để đảm bảo cùng input → cùng output ngay cả khi Formula được đánh giá
nhiều lần trong một Turn.

---

# 19. Non-goals

Rule Execution Model không định nghĩa:

* Rule syntax           (PRD-010)
* Formula syntax        (PRD-009)
* Turn phases           (PRD-005)
* State schema          (PRD-006)
* Query / Selector      (PRD-013)
* Action Resolution     (PRD-012)
* Save format
* Concurrency

Không có thread. Không có async. Runtime là single-threaded.

---

# 20. Tiêu chí hoàn thành

Rule Execution Model được xem là hoàn thành khi:

* Cùng Turn Start Snapshot + cùng Rule set → cùng State sau Commit.
* Mọi conflict giữa Rule cùng Variable được resolve theo §10 mà không
  phụ thuộc thứ tự LLM đọc file.
* Rollback giữ State invariant mà không mất tất cả mutation.
* Builder phát hiện read-after-write, ambiguous ordering, và conflict
  không giải quyết được.
* Turn History đủ để replay Turn deterministic.
