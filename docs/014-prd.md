# PRD-014 — Documentation Schema

**Project:** Markdown Game Runtime (MGR)

**Version:** 1.0

**Status:** Draft

---

# 1. Mục tiêu

Documentation Schema định nghĩa các block **mô tả ý định** của mỗi
declaration — `Purpose`, `Failure`, `Notes`, `Examples`.

PRD-008 §15a đã định nghĩa **execution schema** — cái Runtime CẦN để
thực thi (Kind, Trigger, Precondition, Effect, Priority, Phase,
Attributes, Visibility, ...).

PRD-008 §15a chưa định nghĩa **documentation schema** — cái LLM và
người CẦN để reasoning ổn định.

Documentation Schema lấp phần đó.

Không có PRD này, hai game có execution shape giống nhau nhưng
document shape khác nhau — LLM phải học lại structure mỗi Turn, và
người đọc không biết một Rule "để làm gì".

---

# 2. Triết lý

Documentation Schema phải:

* nhất quán trên mọi Kind
* không bắt buộc khi thực sự không có gì để nói
* phân biệt WARNING (đủ tốt) và ERROR (bắt buộc)
* không sinh side effect trong Runtime
* Markdown-native

Documentation Schema không phải:

* comment
* changelog
* commit message
* linting toàn văn xuôi

Documentation Schema là bốn block bổ sung vào mỗi declaration khi phù
hợp.

---

# 3. Kiến trúc

Bốn block bổ sung.

```text
Purpose      — vì sao khai báo này tồn tại; 1–3 câu
Failure      — điều gì xảy ra khi Precondition không hợp lệ
Notes        — bối cảnh / lịch sử / caveat
Examples     — ví dụ ngắn về input hoặc kết quả
```

Áp dụng cho các Kind của PRD-008 §15a — mỗi Kind có bảng riêng
(xem §5).

Runtime **không** đọc Documentation blocks. Builder đọc để lint. LLM
đọc để reasoning.

---

# 4. Levels

Documentation Schema Validator phát ra tại ba mức.

```text
ERROR    — bắt buộc; build fail nếu thiếu
WARNING  — khuyến nghị; build vẫn pass; hiển thị trong CLI
IGNORE   — hoàn toàn optional; không kiểm tra
```

Mặc định:

* `Purpose`  → ERROR trên Rule / Action / Event.
* `Purpose`  → WARNING trên Entity / Formula / Variable / Query.
* `Failure` → WARNING trên Rule Transformation / Rule Guard / Action.
* `Notes`   → IGNORE.
* `Examples` → IGNORE.

Author có thể suppress một mức xuống bằng comment:

```text
<!-- doc-suppress: Purpose (moved to metadata.md) -->
```

Suppress không được lạm dụng. Builder log số lần suppress.

# 5. Schema per Kind

## 5.1. Variable

```text
Variable <Name>
Visibility: ...
Purpose:               (WARNING)
<1–3 câu>
Notes:                 (IGNORE)
<...>
```

## 5.2. Entity

```text
Entity <Name>
Kind: ...
Attributes: ...
Purpose:               (WARNING)
<vì sao Entity tồn tại>
Notes:                 (IGNORE)
```

## 5.3. Formula

```text
Formula <Name>
<expression>
Purpose:               (WARNING)
<công thức mô tả cái gì>
Notes:                 (IGNORE)
```

## 5.4. Rule

```text
Rule <Name>
Kind: ...
Trigger: ...
Precondition: ...
Effect: ...
Purpose:               (ERROR)
<vì sao Rule tồn tại>
Failure:               (WARNING nếu Kind = Transformation | Guard)
<mô tả điều gì xảy ra khi Precondition sai>
Notes:                 (IGNORE)
Examples:              (IGNORE)
```

## 5.5. Event

```text
Event <Name>
Phase: ...
Trigger: ...
Effect: ...
Purpose:               (ERROR)
<mục đích Event>
Notes:                 (IGNORE)
```

## 5.6. Action

```text
Action <Name>
Intent: ...
Parameters: ...
Preconditions: ...
Purpose:               (ERROR)
<mục đích Action>
Failure:               (WARNING)
<mô tả điều gì xảy ra khi Preconditions sai>
Notes:                 (IGNORE)
Examples:              (IGNORE)
```

## 5.7. Auto Action

```text
Auto Action <Name>
Fires When: ...
Purpose:               (ERROR)
<vì sao Auto Action tồn tại>
Notes:                 (IGNORE)
```

## 5.8. Query

```text
Query <Name>
<expression>
Purpose:               (WARNING)
Notes:                 (IGNORE)
```

# 6. Purpose — Nội dung

Purpose phải:

* 1–3 câu
* không lặp lại tên declaration
* không mô tả cú pháp ("This is a Guard rule")
* mô tả **tại sao** (intent) hoặc **khi nào** (context)

Đúng:

```text
Purpose:
Prevent negative Money after a purchase. Applied to every Buy Action
that spends cash.
```

Sai:

```text
Purpose:
CanBuyLemons is a guard rule.
```

Anti-pattern: **Purpose lặp tên**. Validator emit
`DOCUMENTATION_PURPOSE_TAUTOLOGY` khi Purpose chỉ chứa từ trong tên
declaration.

---

# 7. Failure — Nội dung

Failure mô tả điều gì xảy ra khi Rule/Action không match Precondition.

Đúng:

```text
Failure:
Reject the action. State does not change. Response describes which
resource is insufficient.
```

Không bắt buộc mô tả toàn bộ error handling — chỉ nêu ngữ nghĩa
gameplay-visible.

Nếu Failure không cần (Rule Trigger fire theo Event và không có concept
"fail"), bỏ block. Validator không cảnh báo.

---

# 8. Notes — Nội dung

Notes là nơi cho:

* lịch sử (rule này ra đời từ vụ playtest nào)
* caveat (edge case runtime không tự xử lý)
* pointer sang doc khác

Không dùng Notes để chứa Purpose ẩn. Nếu bạn định viết Purpose vào
Notes để né validator → nên đọc lại §6 và viết Purpose đúng cách.

---

# 9. Examples — Nội dung

Examples là block hoàn toàn optional.

Format khuyến nghị: `Input → Output` hoặc `State → State`.

```text
Examples:
Input: Buy Lemons quantity=5
Output: Money -= 25, Lemons += 5
```

Nhiều Examples được liệt kê tuần tự.

---

# 10. Validator behavior

Documentation Schema Validator (mở rộng của Section Schema Validator
PRD-008 §15a) phát ra:

```text
DOCUMENTATION_PURPOSE_MISSING    — thiếu Purpose (ERROR hoặc WARNING theo §4)
DOCUMENTATION_FAILURE_MISSING    — thiếu Failure (WARNING)
DOCUMENTATION_PURPOSE_TAUTOLOGY  — Purpose lặp lại tên
DOCUMENTATION_BLOCK_EMPTY        — block heading có, content trống
```

WARNING không làm build fail. ERROR làm build fail.

CLI hiển thị:

```text
✓ Validate — 0 errors, 3 warnings
```

Với `--strict` flag, WARNING được nâng lên ERROR.

---

# 11. Interop với PRD-008 §15a

PRD-014 mở rộng §15a — không thay thế.

Section Schema (§15a) chạy trước:

1. Check block bắt buộc (Kind, Trigger, Effect, ...).
2. Check block cấm (Effect trên Guard, ...).
3. Check duplicate block.

Documentation Schema chạy sau:

4. Check Purpose theo §4.
5. Check Failure theo §4.
6. Check tautology.
7. Check block empty.

Bước 1-3 emit ERROR. Bước 4-7 emit theo level trong §4.

---

# 12. Interop với PRD-004 Determinism

Documentation Blocks không được đưa vào Prompt Specification runtime
context nếu chỉ chứa metadata.

Nhưng Purpose và Failure SẼ được đưa vào Prompt Specification vì LLM
đọc để reasoning. Notes và Examples được đưa vào **tuỳ** cấu hình
Builder (`docComments: full | minimal | strip`).

Mặc định: `minimal` — chỉ Purpose và Failure vào Prompt.

Điều này giữ deterministic: cùng Documentation Blocks → cùng Prompt
output.

---

# 13. Formatting

Purpose / Failure / Notes / Examples ở dạng plain markdown.

Không cần bullet, không cần bold. Câu văn thường.

Cấm nhúng directive `@` trong Documentation Blocks.

Cấm nhúng Formula, Rule, Action declaration trong Documentation Blocks.

Nếu cần cross-reference: `see rules.md::CanBuyLemons` (plain text).

---

# 14. CLI

`mgr validate` mặc định emit ERROR + WARNING.

`mgr validate --quiet` chỉ emit ERROR.

`mgr validate --strict` nâng WARNING thành ERROR.

`mgr build` bị ảnh hưởng bởi cùng flag.

CLI phải summarize: `X errors, Y warnings`.

---

# 15. Documentation coverage

Builder có thể emit report:

```text
Coverage:
Rule       12/12 documented (100%)
Action      6/6 documented (100%)
Event       5/5 documented (100%)
Entity      3/3 documented (100%)
Formula     2/3 documented (67%)
```

Chỉ thông tin. Không fail build. Cho phép project tracking chất lượng
tài liệu qua thời gian.

---

# 16. Non-goals

Documentation Schema không định nghĩa:

* Prompt formatting               (PRD-003)
* Rule execution                  (PRD-011)
* Formula grammar                 (PRD-009)
* Turn phases                     (PRD-005)
* i18n của documentation
* Auto-generated documentation từ code
* Grammar / spell check
* Version control ràng buộc

Documentation Schema chỉ là bộ block bổ sung + validator level.

---

# 17. Author guidance

Khuyến nghị viết Purpose ngay khi tạo declaration.

Không dời Purpose sang comment. Comment không được validator kiểm.

Không giữ Purpose ngắn hơn cần thiết. 1 câu tốt hơn 0. 3 câu tốt hơn
1 khi có edge case.

Không viết Purpose bằng ngôn ngữ mơ hồ ("handles the buying"). Viết
cụ thể ("Rejects Buy Lemons when Money < Quantity × 5, preventing
negative balance").

---

# 18. Backwards compatibility

Documentation Schema là mở rộng, không breaking:

* Game hiện tại không có Purpose → build với WARNING/ERROR tuỳ Kind.
* Suppress được qua comment (§4).
* Chuyển đổi hoàn toàn optional cho các game beta.

Reference Games PHẢI tuân thủ đầy đủ để làm gương.

---

# 19. Interaction với PRD-015 Formula Body

PRD-015 (nếu có) sẽ chuẩn hoá Formula body. Purpose block của Formula
vẫn thuộc PRD-014.

Formula body và Purpose không nên overlap. Body là cú pháp; Purpose là
ngữ nghĩa.

---

# 20. Tiêu chí hoàn thành

Documentation Schema được xem là hoàn thành khi:

* Mọi Rule/Action/Event có Purpose.
* Failure được viết cho Rule Transformation và Action khi có ngữ nghĩa
  fail thực sự.
* Validator phân biệt ERROR và WARNING theo §4.
* CLI hiển thị số error và warning riêng.
* Reference Games (Lemonade, Hangman, ...) đạt 100% coverage Purpose.
* PRD tương thích ngược với PRD-008 §15a.
