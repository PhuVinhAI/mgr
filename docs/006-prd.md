# PRD-006 — State System

**Project:** Markdown Game Runtime (MGR)

**Version:** 1.1

**Status:** Draft

---

# 1. Mục tiêu

State System định nghĩa mô hình trạng thái chuẩn của mọi game được xây dựng bằng MGR.

State không phụ thuộc thể loại game.

State không phụ thuộc UI.

State không phụ thuộc Game Package.

Runtime chỉ thao tác trên State thông qua các quy tắc được định nghĩa trong Specification.

---

# 2. Triết lý

State là **nguồn sự thật duy nhất (Single Source of Truth)**.

Mọi quyết định của Runtime phải dựa trên State.

Không được suy diễn bằng Narrative.

Không được "đoán" trạng thái.

Nếu không tồn tại trong State hoặc Rule, Runtime không được coi là tồn tại.

---

# 3. Mô hình

State được xem như một Snapshot của toàn bộ thế giới tại một thời điểm.

```text id="5wqvkh"
Turn 18

↓

State Snapshot

↓

Player Action

↓

State Snapshot

Turn 19
```

Mỗi Turn sinh đúng một Snapshot mới.

# 4. Thành phần của State

State gồm nhiều Domain.

Ví dụ:

```text id="mcrj1r"

World

NPC

Resources

Inventory

Flags

Variables

Timers

Relationships

Quest

History
```

Game có thể thêm Domain mới.

Runtime không giới hạn số lượng Domain.

# 5. Entity

Mọi đối tượng trong State đều là Entity.

Ví dụ:

```text id="imxwr6"
Player

Company

Employee

Factory

Item

Monster
```

Entity phải có định danh duy nhất trong phạm vi game.

# 5a. Transient Entity

State phân biệt hai loại Entity.

```text
Persistent Entity   — tồn tại trong Snapshot, kéo dài nhiều Turn
Transient Entity    — chỉ tồn tại trong một Turn hoặc một Phase
```

Ví dụ Transient:

```text
Cup                 (bán trong Simulation, không lưu)
Combat Round        (tồn tại trong Simulation, không lưu)
Marketing Impression(sinh trong Post Event, không lưu)
```

Transient Entity:

* KHÔNG được Commit vào Snapshot.
* KHÔNG xuất hiện trong Turn History (PRD-005 §19).
* Có thể có Property và Formula.
* Phải khai báo Lifetime (Phase, Turn).

Game Package khai báo:

```text
Entity Cup
Kind: Transient
Lifetime: Simulation Phase
```

Persistent Entity mặc định. Không cần khai báo Kind.

Runtime tạo Transient Entity khi Rule yêu cầu, hủy khi hết Lifetime.

Sai Lifetime → Build fail.

Ví dụ:

```text id="zjlwmk"
Members
```


# 11. Hidden State

State có thể có hai lớp.

```text id="r6dbr8"
Public State

Hidden State
```

Hidden State chỉ Runtime được phép sử dụng.

Ví dụ:

* AI Strategy
* Future Events
* Secret Values
* Hidden Map

Player không được nhìn thấy Hidden State.

# 11a. Visibility Declaration

Game Package phải khai báo Visibility rõ ràng cho mỗi Variable, Property
và Collection.

Cú pháp chuẩn:

```text
Variable Money
Visibility: Public

Variable TodayDemand
Visibility: Hidden

Variable Reserve
Visibility: Private
```

Ba mức duy nhất:

```text
Public    — Player nhìn thấy trong UI
Private   — Runtime dùng, tiết lộ khi Rule cho phép
Hidden    — không bao giờ tiết lộ trực tiếp
```

Visibility mặc định là Public.

Property và Collection dùng cùng cú pháp.

Builder ghi Visibility vào Prompt Specification (PRD-003 §9a) để Runtime
biết cách render.

Runtime KHÔNG được suy diễn Visibility. Nếu không khai báo, Builder
mặc định Public và ghi cảnh báo (không fail).

Vi phạm Visibility trong UI → Runtime Rule violation (PRD-004).

---

# 12. State Mutation

State chỉ được thay đổi trong State Commit.

Không được thay đổi:

* Narrative
* UI
* Validation
* Rendering

Mọi Mutation phải đi qua Turn Lifecycle.

---

# 13. State Validation

Sau mỗi Commit.

Runtime phải đảm bảo:

* Entity hợp lệ.
* Collection hợp lệ.
* Variable hợp lệ.
* Relationship hợp lệ.
* Không có dữ liệu mâu thuẫn.

Nếu Validation thất bại.

Commit phải bị hủy.

---

# 14. State Invariants

Game Package có thể định nghĩa các bất biến.

Ví dụ:

```text id="yg8ek8"
Money >= 0

Population >= 0

HP <= MaxHP
```

Runtime phải luôn duy trì các bất biến này.

---

# 15. Derived State

Một số giá trị có thể được suy ra từ State khác.

Ví dụ:

```text id="jlwmvu"
Net Worth

Average Salary

Combat Power
```

Derived State không được lưu trực tiếp nếu có thể tính lại.

Runtime chỉ tính khi cần.

---

# 16. State Visibility

Mỗi Property có mức hiển thị.

```text id="oq91ga"
Public

Private

Hidden
```

Public:

Player nhìn thấy.

Private:

Chỉ Runtime sử dụng nhưng có thể tiết lộ khi Rule cho phép.

Hidden:

Không bao giờ hiển thị trực tiếp.

---

# 17. State History

Runtime phải duy trì lịch sử State theo Turn.

```text id="gkz70p"
Turn 1

↓

Turn 2

↓

Turn 3

↓

Turn 4
```

History chỉ đọc.

Không được sửa Turn cũ.

---

# 18. Serialization

State phải có thể biểu diễn hoàn toàn bằng Markdown.

Builder và Runtime phải sử dụng cùng một quy ước.

Mục tiêu:

* dễ đọc
* dễ debug
* dễ kiểm tra
* độc lập với nền tảng

Không phụ thuộc JSON trong Prompt Specification.

---

# 19. State Ownership

Chỉ Runtime có quyền:

* tạo
* sửa
* xóa

State.

Player chỉ gửi Action.

Game Package chỉ định nghĩa Schema và Rule.

# 20. Non-goals

State System không định nghĩa:

* UI
* Save File
* Multiplayer
* Networking
* Formula Engine
* Combat System
* Economy

Đây là trách nhiệm của các PRD khác.

---

# 21. Tiêu chí hoàn thành

State System được xem là hoàn thành khi:

* Có mô hình State thống nhất cho mọi game.
* Hỗ trợ Entity, Collection, Variable và Relationship.
* Hỗ trợ Persistent và Transient Entity.
* Hỗ trợ Public, Private, Hidden State với khai báo tường minh.
* Chỉ Runtime được phép thay đổi State.
* Mọi thay đổi đều thông qua Turn Lifecycle.
* State có thể biểu diễn hoàn toàn bằng Markdown và trở thành nền tảng duy nhất cho mọi quyết định của Runtime.
