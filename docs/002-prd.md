# PRD-002 — Markdown Directive System

**Project:** Markdown Game Runtime (MGR)

**Version:** 1.0

**Status:** Draft

---

# 1. Mục tiêu

Directive System là lớp mở rộng của Markdown, cho phép biến Markdown từ tài liệu thuần túy thành ngôn ngữ mô tả Prompt Programming.

Directive không thay đổi cú pháp Markdown.

Directive chỉ bổ sung khả năng tổ chức, tái sử dụng và biên dịch source.

Directive phải:

* Dễ đọc.
* Dễ viết.
* Có thể parse được.
* Có thể mở rộng.
* Không phá vỡ Markdown chuẩn.

---

# 2. Triết lý

Directive không phải scripting language.

Directive không được phép chứa logic tính toán phức tạp.

Directive chỉ mô tả cấu trúc project.

Mọi logic game vẫn được viết bằng Markdown.

---

# 3. Nguyên tắc

Directive luôn bắt đầu bằng:

```text
@
```

Ví dụ:

```text
@import

@section

@include
```

Directive chỉ có tác dụng trong quá trình Build.

LLM sẽ không nhìn thấy Directive.

Builder phải xử lý toàn bộ Directive trước khi sinh Prompt Specification.

---

# 4. Directive Levels

Directive được chia thành bốn nhóm.

## Project

Điều khiển project.

Ví dụ:

* @import
* @include
* @export

---

## Metadata

Mô tả tài liệu.

Ví dụ:

* @title
* @author
* @version
* @description

---

## Structure

Điều khiển cấu trúc.

Ví dụ:

* @section
* @group
* @order

---

## Build

Điều khiển compiler.

Ví dụ:

* @if
* @endif
* @warning
* @deprecated

---

# 5. Directive đầu tiên

Foundation chỉ hỗ trợ hai Directive.

## @import

Nhập nội dung file khác.

Ví dụ:

```text
@import runtime/engine
```

Builder:

* tìm file
* parse
* validate
* chèn vào vị trí hiện tại

---

## @section

Định nghĩa section.

Ví dụ:

```text
@section Runtime
```

Builder dùng để:

* tạo TOC
* validate
* sắp xếp

---

# 6. Cú pháp

Directive luôn nằm trên một dòng riêng.

Ví dụ:

```text
@import runtime/state

@section Runtime
```

Không hỗ trợ:

```text
abc @import runtime
```

---

# 7. Đường dẫn

Đường dẫn luôn tương đối từ thư mục source.

Ví dụ:

```text
@import runtime/state

@import ui/dashboard
```

Không hỗ trợ:

* URL
* HTTP
* Absolute Path

---

# 8. Import Rules

Một file chỉ được import một lần trong cùng một nhánh.

Nếu import trùng:

Builder bỏ qua.

Nếu import vòng:

Builder báo lỗi.

Ví dụ:

```text
A

↓

B

↓

C

↓

A
```

Build phải thất bại.

---

# 9. Section Rules

Tên Section phải duy nhất.

Ví dụ:

```text
@section Runtime
```

Không được có:

```text
@section Runtime
```

ở hai nơi khác nhau.

---

# 10. Error Handling

Ví dụ:

```text
Unknown directive

@imports runtime
```

Builder:

```text
Unknown directive "@imports"

Did you mean:

@import
```

---

# 11. Reserved Directives

Các Directive sau được dành trước.

Chưa cần hỗ trợ ở phiên bản hiện tại.

```text
@include

@define

@const

@if

@elseif

@else

@endif

@warning

@deprecated

@plugin

@feature

@using

@override

@replace

@remove

@insert-before

@insert-after

@toc

@export
```

Không được cho phép developer tự định nghĩa các tên này.

---

# 12. Parser

Parser phải:

* nhận biết Directive
* lấy tên
* lấy tham số
* kiểm tra cú pháp
* sinh AST

Ví dụ:

```text
@import runtime/state
```

AST:

```text
Directive

type:
import

argument:
runtime/state
```

---

# 13. Validation

Builder phải kiểm tra:

* Directive tồn tại.
* Thiếu tham số.
* Sai cú pháp.
* Sai đường dẫn.
* Import vòng.
* Section trùng.

Nếu có lỗi:

Build dừng ngay.

---

# 14. Khả năng mở rộng

Directive mới phải có thể được thêm mà không thay đổi parser hiện tại.

Parser phải được thiết kế theo Registry.

Ví dụ:

```text
ImportDirective

SectionDirective

FutureDirective
```

Parser không được hard-code từng Directive.

---

# 15. Non-goals

Phiên bản này không hỗ trợ:

* Macro
* Variables
* Expressions
* Loops
* Arithmetic
* Runtime Evaluation
* User-defined Directive

---

# 16. Tiêu chí hoàn thành

Directive System được coi là hoàn thành khi:

* Parser nhận diện được Directive.
* Hỗ trợ @import.
* Hỗ trợ @section.
* Phát hiện lỗi cú pháp.
* Phát hiện import vòng.
* Phát hiện section trùng.
* Có thể mở rộng thêm Directive mà không cần viết lại parser.
