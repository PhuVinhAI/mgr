# PRD-010 — Rule Language

**Project:** Markdown Game Runtime (MGR)

**Version:** 1.0

**Status:** Draft

---

# 1. Mục tiêu

Rule Language định nghĩa cấu trúc để mô tả luật chơi trong MGR.

PRD-008 §8 đã nói Rule là trái tim của Game Package.

PRD-008 §8 chưa nói Rule được viết như thế nào.

Rule Language lấp phần đó.

---

# 2. Triết lý

Rule phải:

* rõ ràng
* có cấu trúc cố định
* machine-readable
* Markdown-native
* dùng Formula (PRD-009) làm biểu thức

Rule không phải:

* văn bản tự do
* code
* JSON
* prompt

Rule là một cấu trúc được đặc tả bằng chính Markdown.

---

# 3. Kiến trúc

Mỗi Rule gồm 4 khối.

```text
Rule <Name>

├── Trigger        (khi nào Rule được xét)
├── Precondition   (điều kiện để Rule áp dụng)
├── Effect         (thay đổi State nếu áp dụng)
└── Priority       (thứ tự nếu xung đột)
```

---

# 4. Rule kinds

MGR định nghĩa ba loại Rule.

```text
Guard          — chặn Action nếu Precondition sai
Transformation — áp dụng Effect khi Precondition đúng
Trigger        — kích hoạt Effect khi Trigger fire
```

Game Package chọn loại phù hợp.

Một Rule chỉ có một loại.

---

# 5. Rule declaration

Cú pháp chuẩn.

```text
Rule <Name>

Kind: <Guard | Transformation | Trigger>

Trigger:
<mô tả khi nào Rule chạy>

Precondition:
<Formula trả Boolean>

Effect:
<danh sách mutation>

Priority:
<Integer, lớn hơn = ưu tiên hơn>
```

Priority mặc định = 0.

---

# 6. Trigger

Trigger cho biết Rule chạy ở phase nào của Turn Lifecycle.

```text
On Action(<action name>)
On Pre Event
On Post Event
On State Commit
Always
```

Nhiều Trigger được nối bằng OR.

Rule không có Trigger → được xử lý như Always trong pha Simulation.

---

# 7. Precondition

Precondition là một Formula trả Boolean (PRD-009 §7).

Ví dụ:

```text
Precondition:
Money ≥ Quantity × 5
```

Nhiều điều kiện dùng AND / OR trong cùng Formula.

Precondition sai → Rule không áp dụng.

Với Guard: Precondition sai = từ chối Action.

---

# 8. Effect

Effect là danh sách các mutation lên State.

Mỗi mutation có dạng.

```text
<Variable> <Op> <Formula>
```

Với `Op`:

```text
:=   gán
+=   cộng thêm
-=   trừ đi
×=   nhân với
÷=   chia cho
```

Ví dụ:

```text
Effect:
Money -= Quantity × 5
Lemons += Quantity
```

Effect chỉ chạy trong Simulation Phase hoặc Event Phase (PRD-005 §14
Side Effects).

---

# 9. Multiple Effects

Nhiều mutation trong một Effect được thực hiện đồng thời (atomic) theo
PRD-005 §12.

Không được đọc giá trị đã bị mutation trong cùng Effect.

Sai:

```text
Effect:
Money += Bonus
Reputation += Money × 0.01   # đọc Money mới → sai
```

Đúng:

```text
Effect:
Money += Bonus
Reputation += (Money + Bonus) × 0.01
```

---

# 10. Priority

Nhiều Rule có thể cùng khớp Trigger.

```text
Priority: 10   → chạy trước
Priority: 0    → chạy sau
```

Cùng Priority → theo thứ tự xuất hiện trong Game Package.

Priority chỉ trong cùng loại Rule.

PRD-005 §8 vẫn giữ ưu tiên toàn cục:
System → Runtime → Game → Event.

---

# 11. Rule composition

Rule không được gọi Rule khác trực tiếp.

Rule chỉ mô tả điều kiện và hiệu ứng.

Nếu cần logic phức tạp, tách thành nhiều Rule với Trigger phù hợp.

Điều này giữ Rule ở dạng flat, không đệ quy, không side-effect ngầm.

---

# 12. Guard Rule pattern

Guard chạy trong Validation Phase (PRD-005 §5).

```text
Rule CanBuyLemons

Kind: Guard

Trigger:
On Action(Buy Lemons)

Precondition:
Money ≥ Quantity × 5
```

Nếu Precondition sai → Action bị từ chối.

Guard không có Effect.

---

# 13. Transformation Rule pattern

Transformation chạy trong Simulation Phase.

```text
Rule ApplyBuyLemons

Kind: Transformation

Trigger:
On Action(Buy Lemons)

Precondition:
Money ≥ Quantity × 5

Effect:
Money -= Quantity × 5
Lemons += Quantity
```

Cùng Action có thể có nhiều Transformation. Chạy theo Priority.

---

# 14. Trigger Rule pattern

Trigger chạy trong Pre Event hoặc Post Event.

```text
Rule HeatWaveEffect

Kind: Trigger

Trigger:
On Pre Event
Event.Name = Heat Wave

Precondition:
true

Effect:
IcePerCup := 2
```

Trigger Rule không gắn với Action cụ thể.

---

# 15. Named Formulas trong Rule

Rule có thể dùng Named Formula (PRD-009 §14).

```text
Rule DailyDemand

Kind: Trigger

Trigger:
On Pre Event

Effect:
TodayDemand := BaseDemand × WeatherModifier
```

Nếu `BaseDemand` chưa được định nghĩa → Build fail.

---

# 16. Rule validation

Builder kiểm tra:

* Rule có Kind hợp lệ.
* Guard không có Effect.
* Transformation và Trigger có ít nhất một mutation trong Effect.
* Mọi Reference resolve.
* Priority là Integer.
* Trigger hợp lệ với Kind.
* Precondition trả Boolean.

Sai bất kỳ → Build fail.

---

# 17. Rule isolation

Một Rule không được biết Rule khác.

Không có `import`, `extend`, `include` giữa Rule.

Rule chỉ tương tác qua State.

---

# 18. Debug support

Runtime phải ghi vào Turn History (PRD-005 §19):

* Rule nào đã kích hoạt
* Rule nào bị Precondition từ chối
* Effect nào đã Commit

Điều này cho phép replay và audit.

---

# 19. Non-goals

Rule Language không định nghĩa:

* Formula syntax     (PRD-009)
* Turn phases        (PRD-005)
* State schema       (PRD-006)
* Event lifecycle    (PRD-005)
* UI                 (PRD-007)
* AI behavior beyond Rule
* Multiplayer sync

---

# 20. Tiêu chí hoàn thành

Rule Language được xem là hoàn thành khi:

* Mọi luật chơi có thể biểu diễn bằng Guard / Transformation / Trigger.
* Không cần dùng ngôn ngữ ngoài Markdown + Formula.
* Builder kiểm tra được cấu trúc và Reference của Rule.
* Runtime đánh giá Rule theo Priority deterministic.
* Rule tương thích với Turn Lifecycle (PRD-005) và Formula System
  (PRD-009).
