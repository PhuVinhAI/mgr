# PRD-003 — Prompt Specification Format (PSF)

**Project:** Markdown Game Runtime (MGR)

**Version:** 1.1

**Status:** Draft

---

# 1. Mục tiêu

Prompt Specification Format (PSF) là chuẩn tài liệu đầu ra của MGR.

Builder phải luôn sinh ra đúng một Prompt Specification.

LLM chỉ đọc Prompt Specification.

LLM không biết project gồm bao nhiêu file.

LLM không biết Directive.

LLM không biết Builder.

---

# 2. Triết lý

PSF là một tài liệu đặc tả (Specification Document), không phải một prompt ngẫu nhiên.

PSF phải:

* Có cấu trúc ổn định.
* Dễ đọc.
* Dễ điều hướng.
* Dễ mở rộng.
* Độc lập với game.

---

# 3. Nguyên tắc

Builder phải luôn sinh cùng một cấu trúc.

Không được thay đổi thứ tự Section theo từng project.

Điều này giúp:

* tăng tính ổn định
* dễ debug
* dễ mở rộng
* tương thích ngược

---

# 4. Kiến trúc

Prompt Specification gồm ba tầng.

```text
System Layer

↓

Runtime Layer

↓

Game Layer
```

---

# 5. System Layer

System Layer định nghĩa cách LLM hoạt động.

Bao gồm:

```text
SYSTEM

IDENTITY

MISSION

PRIORITIES

GLOBAL CONSTRAINTS
```

Ví dụ:

```md
# SYSTEM

You are the runtime.

Never act as an assistant.
```

---

# 6. Runtime Layer

Runtime Layer mô tả engine.

Bao gồm:

```text
RUNTIME

TURN LOOP

STATE MACHINE

VALIDATION

MEMORY MODEL

OUTPUT RULES
```

Runtime Layer không chứa gameplay.

---

# 7. Game Layer

Game Layer mô tả game.

Bao gồm:

```text
GAME

WORLD

ENTITIES

RULES

EVENTS

START STATE

WIN CONDITION

LOSE CONDITION
```

Game Package sẽ sinh nội dung này.

---

# 8. UI Contract

Prompt Specification luôn phải có UI Contract.

Ví dụ:

```text
UI CONTRACT
```

Trong phần này mô tả:

* Dashboard
* Sections
* Tables
* Action Menu
* Notifications

LLM phải luôn render đúng theo hợp đồng này.

---

# 9. State Contract

State Contract mô tả:

* biến
* object
* collection
* relationship

Ví dụ:

```text
Money

Employees

Projects

Flags
```

State Contract không chứa giá trị hiện tại.

# 9a. Visibility Preservation

State Contract phải giữ Visibility (PRD-006 §11a).

Mỗi Variable, Property, Collection đều mang một trong ba nhãn.

```text
[Public]
[Private]
[Hidden]
```

Ví dụ trong Prompt Specification:

```text
Money             [Public]
Reputation        [Public]
TodayDemand       [Hidden]
Reserve           [Private]
```

Runtime đọc nhãn để:

* Public   → render trong Dashboard và Snapshot.
* Private  → chỉ tiết lộ khi Rule cho phép.
* Hidden   → không bao giờ render trực tiếp.

Bundler bắt buộc phải ghi Visibility. Không ghi = mặc định Public + warning.

LLM KHÔNG được đoán Visibility. Nhãn là quy định.

---

# 10. Output Contract

Output Contract mô tả chính xác format phản hồi.

Ví dụ:

```text
Narrative

↓

Dashboard

↓

Available Actions

↓

Current State

↓

Wait Player Input
```

Mọi lượt chơi đều phải tuân theo Output Contract.

---

# 11. Section Order

Builder luôn sinh đúng thứ tự sau:

```text
SYSTEM

IDENTITY

MISSION

GLOBAL CONSTRAINTS

RUNTIME

TURN LOOP

STATE MACHINE

MEMORY MODEL

VALIDATION

UI CONTRACT

STATE CONTRACT

GAME

WORLD

ENTITIES

RULES

EVENTS

START STATE

OUTPUT CONTRACT

FIRST TURN
```

Không được thay đổi.

---

# 12. Section Identity

Mỗi Section phải có ID nội bộ.

Ví dụ:

```text
SYSTEM

id: system
```

Builder sử dụng ID để:

* merge
* validate
* debug
* override

LLM không cần quan tâm.

---

# 13. Navigation

Builder phải sinh Table of Contents.

Ví dụ:

```text
Table of Contents

1 SYSTEM

2 RUNTIME

3 GAME

4 UI

5 OUTPUT
```

TOC giúp cả con người và LLM định vị tài liệu.

---

# 14. Metadata

Prompt Specification phải có Metadata.

Ví dụ:

```text
Project

Version

Build Date

Compiler Version

Specification Version
```

Metadata chỉ phục vụ debug.

Không ảnh hưởng gameplay.

---

# 15. Prompt Boundary

Builder phải phân tách Section rõ ràng.

Ví dụ:

```text
================================================

SYSTEM

================================================
```

Hoặc:

```text
---

# SYSTEM
```

Không được ghép nội dung thành một khối liên tục.

---

# 16. Prompt Purity

Prompt Specification không được chứa:

* Directive
* Import
* Comment của compiler
* Debug information
* Temporary content

Output chỉ chứa nội dung cuối cùng.

---

# 17. Prompt Stability

Nếu source không đổi:

Output phải giống hệt.

Builder không được:

* đổi thứ tự
* đổi heading
* đổi khoảng trắng quan trọng
* đổi Section

---

# 18. Compatibility

PSF phải độc lập với:

* ChatGPT
* Claude
* Gemini
* Grok
* DeepSeek
* các LLM khác

Prompt Specification không chứa chỉ dẫn dành riêng cho một nền tảng.

Nếu cần tối ưu cho từng LLM, Builder sẽ thực hiện thông qua các profile build trong tương lai, nhưng vẫn phải tuân thủ PSF.

---

# 19. Non-goals

PSF không định nghĩa:

* Gameplay
* Plugin
* Save Format
* Project Structure
* Directive
* CLI

Những phần này thuộc các PRD khác.

---

# 20. Tiêu chí hoàn thành

PSF được coi là hoàn thành khi:

* Builder luôn sinh một Prompt Specification duy nhất.
* Thứ tự các Section luôn cố định.
* Không còn Directive trong output.
* Có TOC và Metadata.
* Có ranh giới Section rõ ràng.
* State Contract giữ đầy đủ Visibility (Public/Private/Hidden).
* Có thể áp dụng cho mọi game theo lượt mà không cần thay đổi định dạng đầu ra.
