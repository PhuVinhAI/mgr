# PRD-007 — UI Contract

**Project:** Markdown Game Runtime (MGR)

**Version:** 1.0

**Status:** Draft

---

# 1. Mục tiêu

UI Contract định nghĩa cách Runtime trình bày thông tin cho người chơi.

UI Contract không mô tả gameplay.

UI Contract không mô tả State.

UI Contract chỉ mô tả giao diện đầu ra bằng Markdown.

Mọi Game Package đều phải tuân thủ cùng một UI Contract.

---

# 2. Triết lý

UI là một View.

UI không chứa logic.

UI không được thay đổi State.

UI chỉ phản ánh State hiện tại.

---

# 3. Nguyên tắc

Runtime luôn render theo Layout cố định.

Không được tự ý thay đổi cấu trúc giữa các Turn.

Player phải luôn biết:

* chuyện gì vừa xảy ra
* trạng thái hiện tại
* mình có thể làm gì tiếp theo

---

# 4. Layout

Mỗi Turn có cùng cấu trúc.

```text id="l7n9rb"
Narrative

↓

Events

↓

Dashboard

↓

Details

↓

Available Actions

↓

Prompt

↓

State Snapshot
```

Không được đổi thứ tự.

---

# 5. Narrative

Narrative mô tả diễn biến.

Narrative:

* ngắn gọn
* đúng Rule
* không tiết lộ Hidden State
* không chứa UI

---

# 6. Events

Nếu Turn có Event.

Runtime hiển thị:

* Event Name
* Effect Summary

Không hiển thị Event nội bộ.

---

# 7. Dashboard

Dashboard là vùng thông tin quan trọng nhất.

Dashboard luôn xuất hiện.

Ví dụ:

```text id="m49ez0"
Money

Health

Energy

Population

Employees

Date
```

Game Package quyết định Dashboard gồm những gì.

Layout vẫn giống nhau.

---

# 8. Detail Panels

Các thông tin mở rộng.

Ví dụ:

```text id="1rrldg"
Inventory

Research

Projects

Army

Relationships
```

Runtime chỉ hiển thị khi cần.

---

# 9. Available Actions

Cuối mỗi Turn.

Runtime luôn liệt kê Action khả dụng.

Ví dụ:

```text id="m9yn4k"
1.

Research

2.

Recruit

3.

Wait
```

Player không bắt buộc phải chọn trong danh sách.

Nhưng Runtime phải luôn cung cấp danh sách.

---

# 10. Prompt

UI luôn kết thúc bằng lời mời nhập Turn tiếp theo.

Ví dụ:

```text id="0np9i8"
What will you do next?
```

Không được tự động xử lý Turn tiếp.

---

# 11. State Snapshot

UI luôn kết thúc bằng State Snapshot.

Snapshot dùng để:

* debug
* save
* load
* replay

Snapshot không phải Narrative.

Snapshot luôn theo chuẩn State System.

---

# 12. Markdown Rules

Runtime chỉ sử dụng Markdown chuẩn.

Cho phép:

* Heading
* Table
* List
* Blockquote
* Bold
* Italic
* Horizontal Rule
* Code Block

Không sử dụng HTML.

Không phụ thuộc CSS.

---

# 13. Stable Layout

Layout phải ổn định.

Ví dụ:

Turn 1 và Turn 500.

Dashboard vẫn ở cùng vị trí.

Action Menu vẫn ở cùng vị trí.

---

# 14. Adaptive Detail

Runtime có thể rút gọn hoặc mở rộng thông tin.

Ví dụ:

Early Game:

Chỉ hiện Dashboard.

Late Game:

Thêm Project Panel.

Nguyên tắc:

Không đổi Layout.

Chỉ thay đổi nội dung.

---

# 15. Hidden Information

Hidden State không được render.

Nếu Rule yêu cầu.

Runtime có thể hiển thị:

```text id="c2h4ul"
???
```

hoặc:

```text id="3g5tqs"
Unknown
```

Không tiết lộ giá trị thật.

---

# 16. Error UI

Nếu Action thất bại.

Runtime không đổi Layout.

Narrative thay bằng lỗi.

Dashboard vẫn hiển thị.

Action Menu vẫn hiển thị.

State Snapshot vẫn hiển thị.

---

# 17. Accessibility

UI phải:

* dễ đọc
* khoảng cách rõ ràng
* tiêu đề nhất quán
* bảng đơn giản
* không lạm dụng biểu tượng

Không phụ thuộc màu sắc.

Không phụ thuộc emoji.

Emoji chỉ là tùy chọn của Game Package.

---

# 18. Renderer Independence

UI Contract mô tả cấu trúc.

Không mô tả cách ChatGPT hiển thị.

Runtime không giả định:

* font
* theme
* dark mode
* mobile
* desktop

Markdown phải hoạt động trên mọi nền tảng.

---

# 19. UI Invariants

Mọi Turn phải đảm bảo:

* Có Narrative.
* Có Dashboard.
* Có Available Actions.
* Có Prompt.
* Có State Snapshot.
* Không có Hidden State.
* Không có thay đổi Layout.

---

# 20. Non-goals

UI Contract không định nghĩa:

* Gameplay
* State Schema
* Formula
* Event Logic
* Animation
* Rich Media
* HTML
* CSS

Các nội dung này thuộc các PRD khác.

---

# 21. Tiêu chí hoàn thành

UI Contract được xem là hoàn thành khi:

* Mọi game MGR đều có cùng cấu trúc giao diện.
* Runtime luôn sinh Markdown hợp lệ.
* Layout ổn định qua mọi Turn.
* Người chơi luôn biết diễn biến, trạng thái và các hành động tiếp theo.
* State Snapshot luôn có mặt để phục vụ Save, Load và Debug.
