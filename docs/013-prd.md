# PRD-013 — Query & Selector System

**Project:** Markdown Game Runtime (MGR)

**Version:** 1.0

**Status:** Draft

---

# 1. Mục tiêu

Query & Selector System định nghĩa cách Rule và Formula **truy vấn**
Collection và tập Entity.

PRD-006 §7 đã nói:

> State supports collections (employees, inventory, cities, projects,
> quests).

PRD-006 §7 chưa nói làm sao để:

* lọc Collection (`Employees where Experience > 80`)
* project ra field (`Projects.Name`)
* aggregation theo điều kiện (`Sum(Employees where Role = Programmer,
  Salary)`)
* sort / limit
* thao tác First / Any / All

Không có Query, Rule phải diễn đạt bằng văn xuôi. LLM đoán. Vi phạm
PRD-004 §4.

Query & Selector System lấp phần đó.

---

# 2. Triết lý

Query phải:

* deterministic
* pure (không mutate)
* stateless (không giữ result giữa Turn)
* readable trong Markdown
* dựa trên Formula (PRD-009)

Không phải:

* SQL
* JavaScript
* prompt "hãy tìm tất cả …"

Query là biểu thức mở rộng của Formula.

---

# 3. Kiến trúc

Query là biểu thức có shape.

```text
<Source> where <Predicate> select <Projection> order by <Key> limit N
```

Từng phần optional trừ Source.

Với `<Source>`:

```text
<Collection Name>
<Entity Type Name>
<Named Query Name>
```

Query trả:

* Collection (nếu chỉ có `where`, `order by`, `limit`)
* Scalar (nếu có `Aggregate(...)` bao ngoài)
* Boolean (nếu là `Any` / `All`)

---

# 4. Selectors trên Collection

Cho Collection tên `Employees`, mỗi phần tử là Entity có Properties.

```text
Employees where Experience > 80
```

Trả Collection con của Employees.

```text
Employees.Salary
```

Là projection — trả List of Rational (không phải Collection Entity).

```text
Employees[0]
```

Truy cập theo index — **cấm** vì không deterministic khi Collection
chưa order.

Muốn lấy phần tử đầu:

```text
First(Employees order by Experience descending)
```

---

# 5. Predicate

Predicate là Formula trả Boolean trên một Entity biến địa phương.

```text
Employees where Salary ≥ 1000 AND Role = Programmer
```

Trong predicate, tên Property của Entity đóng vai Reference. Nếu tên
Property xung đột với biến State toàn cục, Property của Entity thắng.

Cần rõ ràng:

```text
Employees where this.Salary ≥ 1000
```

`this` chỉ Entity đang xét.

---

# 6. Projection

Projection lấy một hoặc nhiều Property.

```text
Employees.Salary                 → List Rational
Employees.Name                   → List String
Employees select (Name, Salary)  → List Tuple(String, Rational)
```

Projection không tạo Entity mới. Nó chỉ trả giá trị.

---

# 7. Aggregation trên Query

PRD-009 §12 đã có `Sum`, `Count`, `Average` trên Collection.

PRD-013 mở rộng cho Query.

```text
Sum(Employees where Role = Programmer, Salary)
Count(Projects where Status = Finished)
Average(Employees.Salary)
Max(Employees, Experience)
Min(Employees, Experience)
```

Cú pháp:

```text
Aggregate(<Source or Query>, <Property or Expression>)
```

Aggregate luôn deterministic theo PRD-009 §12.

---

# 8. Quantifier

Kiểm tra Predicate trên Collection.

```text
Any(Employees where Salary > 1000)
All(Projects where Status = Finished)
None(Enemies where HP > 0)
```

Trả Boolean.

Không đọc phần tử cụ thể — chỉ trả yes/no.

---

# 9. Order & Limit

```text
Employees order by Experience descending
Projects order by Deadline ascending limit 5
```

`order by` bắt buộc trước `limit`.

Nếu không có `order by`, Collection giữ nguyên thứ tự khai báo trong
State — không được tự sắp xếp.

Runtime bằng cách nào cũng phải trả cùng thứ tự cho cùng input.

---

# 10. First / Last

```text
First(<Query>)
Last(<Query>)
```

Trả phần tử đầu / cuối theo thứ tự Query đã sort.

Nếu Query rỗng → trả `Null` và Rule phải xử lý qua `When ... Then` (PRD-009
§9) hoặc Precondition.

---

# 11. Named Query

Query có thể đặt tên và tái sử dụng (giống Named Formula PRD-009 §14).

```text
Query TopEarners
Employees where Salary > 5000 order by Salary descending limit 10

Query FinishedProjects
Projects where Status = Finished
```

Sau đó:

```text
Formula PayrollTop
Sum(TopEarners, Salary)
```

Builder resolve tên Query tại Build time. Tham chiếu vòng → Build fail
(giống Formula).

---

# 12. Reference Resolution

Builder kiểm tra:

* Source phải là Collection, Entity type, hoặc Named Query đã declared.
* Property trong Predicate/Projection phải tồn tại trên Entity type.
* Aggregate expression phải trả kiểu số.
* Quantifier expression phải trả Boolean.
* `order by` key phải orderable (Integer, Rational, String).
* `limit N` phải Integer dương.

Sai → Build fail.

---

# 13. Evaluation Semantics

Query đọc State ở thời điểm gọi.

Trong Rule Effect (PRD-011 §7), Query đọc **Turn Start Snapshot** —
không đọc Pending Delta của các Rule khác.

Trong Simulation Formula (PRD-009), Query đọc snapshot đầu Phase.

Query không mutate. Kết quả không được cache qua Turn.

---

# 14. Composition

Query có thể lồng nhau qua Named Query.

Không hỗ trợ `join`, `group by`, subquery lồng vào predicate.

Nếu cần group / join → dùng nhiều Named Query cộng với Rule tính toán
trung gian ghi vào biến State phụ.

Điều này giữ Query flat và dễ phân tích tĩnh.

---

# 15. Determinism

Cùng State + cùng Query → cùng result set (bao gồm cùng thứ tự).

Không được dùng ngẫu nhiên trong Query.

Nếu cần chọn ngẫu nhiên từ Collection:

```text
Uniform(1, Count(Employees))
```

kết hợp với Query để lấy phần tử tại index đó (trong biến tạm) — nhưng
cách này bị chặn bởi §4 (cấm index truy cập). Đúng cách là Named Query
sort theo hash Property, kết hợp `Uniform`.

Chi tiết implementation tuỳ Builder, nhưng interface luôn deterministic.

---

# 16. Interaction với Rule

Query được dùng trong:

* Precondition — `Any(Enemies where HP > 0)`
* Effect vế phải — `Money += Sum(Sales, Revenue)`
* Named Formula body
* Auto Action `Fires When` (PRD-012 §10)

Query **không được** đứng ở vế trái Assignment. Không ghi vào Collection
qua Query.

Muốn mutate Collection → dùng Action + Rule ghi rõ Entity nào.

---

# 17. Debug

Turn History (PRD-005 §19) ghi:

* Query name (nếu Named).
* Source size.
* Predicate hit count.
* Sample kết quả (theo policy, không phải toàn bộ).

Cho phép audit tại sao Rule chọn Entity X mà không phải Entity Y.

---

# 18. Non-goals

Query & Selector System không định nghĩa:

* Rule execution       (PRD-011)
* Formula syntax       (PRD-009)
* Rule syntax          (PRD-010)
* State schema         (PRD-006)
* Persistence / index
* SQL compatibility
* Joins
* Group by
* Subquery trong Predicate
* Materialized view

Chỉ đủ cho gameplay logic. Nếu cần nhiều hơn → chia thành nhiều Named
Query.

---

# 19. Interoperability

Query xuất hiện trong 4 nơi và tuân thủ đồng bộ:

```text
Rule Precondition   PRD-010 §7
Rule Effect RHS     PRD-010 §8
Formula body        PRD-009 §14
Auto Action guard   PRD-012 §10
```

Không có ngữ cảnh nào khác được phép nhúng Query.

---

# 20. Tiêu chí hoàn thành

Query & Selector được xem là hoàn thành khi:

* Mọi truy vấn Collection và tập Entity có thể biểu diễn không cần
  văn xuôi.
* Result deterministic với cùng State.
* Không cache qua Turn.
* Không mutate State.
* Compatible với Formula (PRD-009), Rule (PRD-010), Execution Model
  (PRD-011), Action Resolution (PRD-012).
* Builder kiểm tra Reference và Type của Query.
