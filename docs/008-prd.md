# PRD-008 — Game Package Specification (GPS)

**Project:** Markdown Game Runtime (MGR)

**Version:** 1.2

**Status:** Draft

---

# 1. Mục tiêu

Game Package Specification (GPS) định nghĩa chuẩn để một trò chơi được tích hợp vào MGR.

Runtime không biết game là gì.

Builder không biết game là gì.

Game chỉ là một Package tuân thủ Specification này.

Điều này cho phép cùng một Runtime chạy nhiều game khác nhau mà không cần thay đổi Runtime.

---

# 2. Triết lý

Game Package chỉ mô tả game.

Game Package không mô tả Runtime.

Game Package không chứa compiler.

Game Package không thay đổi Turn Lifecycle.

Game Package chỉ cung cấp dữ liệu và luật chơi.

---

# 3. Kiến trúc

```text
Game Package

├── Metadata
├── World
├── Rules
├── State Schema
├── Entities
├── Events
├── UI Configuration
└── Start Scenario
```

Builder sẽ hợp nhất Game Package với Runtime Specification để tạo Prompt Specification cuối cùng.

---

# 4. Metadata

Mỗi Game Package phải khai báo:

* Name
* Version
* Author
* Description
* Target Runtime Version
* Package Version

Ví dụ:

```text
Name:
Game Dev Tycoon

Runtime:
1.x

Package:
1.0.0
```

---

# 5. World Definition

Game phải mô tả thế giới.

Ví dụ:

* thời gian
* bối cảnh
* quốc gia
* công nghệ
* luật vật lý
* văn hóa

Runtime chỉ đọc.

Runtime không tự bổ sung World.

---

# 6. State Schema

Game phải khai báo State mà mình sử dụng.

Ví dụ:

```text
Money

Employees

Projects

Research

Fans

Reputation
```

Runtime không tự sinh Variable.

---

# 7. Entity Definition

Game định nghĩa mọi Entity.

Ví dụ:

```text
Player

Company

Employee

Engine

Console
```

Entity phải mô tả:

* thuộc tính
* hành vi
* quan hệ

Runtime chỉ thao tác.

---

# 8. Rules

Rules là trái tim của Game Package.

Rules mô tả:

* điều kiện
* giới hạn
* kết quả

Ví dụ:

```text
Muốn thuê nhân viên

↓

Money >= Salary
```

Runtime phải luôn tuân thủ Rule.

---

# 9. Events

Game định nghĩa Event.

Ví dụ:

* Story
* Random
* Economy
* Disaster
* Festival

Runtime chỉ kích hoạt.

Runtime không tự tạo Event mới.

---

# 10. Actions

Game phải khai báo Action mà người chơi có thể thực hiện.

Ví dụ:

```text
Research

Hire

Build

Travel

Attack
```

Runtime không được tự phát minh Action.

Người chơi có thể nhập tự do, nhưng Runtime phải ánh xạ vào Action hợp lệ hoặc từ chối.

---

# 11. Victory & Failure

Game Package phải định nghĩa:

* Win Condition
* Lose Condition
* Soft Ending
* Hard Ending (nếu có)

Runtime không tự kết thúc trò chơi nếu Package không quy định.

---

# 12. UI Configuration

Game chỉ cấu hình nội dung UI.

Không thay đổi Layout.

Ví dụ:

Dashboard:

```text
Money

Fans

Employees

Current Project
```

Layout vẫn do UI Contract quyết định.

---

# 13. Start Scenario

Game Package phải cung cấp trạng thái khởi đầu.

Bao gồm:

* State ban đầu
* Narrative mở đầu
* Event đầu tiên (nếu có)
* Action đầu tiên

Runtime sử dụng Start Scenario để bắt đầu Turn 1.

---

# 14. Content Isolation

Game Package không được:

* sửa Runtime
* sửa Turn Lifecycle
* sửa UI Contract
* sửa State System

Nếu cần hành vi mới, phải mở rộng thông qua cơ chế Extension hoặc Plugin trong các phiên bản sau.

---

# 15. Package Validation

Builder phải kiểm tra:

* Metadata đầy đủ
* State Schema hợp lệ
* Entity hợp lệ
* Rule hợp lệ
* Event hợp lệ
* Action hợp lệ
* Start Scenario tồn tại
* Section Schema (§15a) tuân thủ

Thiếu bất kỳ thành phần bắt buộc nào thì Build phải thất bại.

---

# 15a. Section Schema

<!-- Added v1.1. Nguồn: gap phát hiện khi triage bản Lemonade v1: nội
     dung tự do trộn với declaration máy đọc, dẫn tới LLM đoán structure.
     Đóng khoảng cách "design doc" vs "specification". -->

Mỗi Kind trong Game Package tuân thủ **schema block cố định**. Mỗi
Kind có một tập block bắt buộc và một tập block optional. Builder
kiểm tra sự có mặt và thứ tự bằng Section Schema Validator.

## 15a.1. Variable

```text
Variable <Name>
Visibility: Public | Private | Hidden
Type: Integer | Rational | Boolean | Enum(...) | String
Default: <literal>
Domain: <Formula trả Boolean, optional>
```

Blocks bắt buộc: `Variable`, `Visibility`.
Optional: `Type` (mặc định Integer), `Default`, `Domain`.

## 15a.2. Entity

```text
Entity <Name>
Kind: Persistent | Transient
Lifetime: <Phase name>   (bắt buộc nếu Kind = Transient)

Attributes:
<danh sách Property>

Behaviour:
<mô tả ngắn, tuỳ chọn — không machine-parsed>

Relationships:
<danh sách quan hệ, optional>
```

Blocks bắt buộc: `Entity`, `Kind`, `Attributes`.
Optional: `Lifetime` (bắt buộc nếu Transient), `Behaviour`, `Relationships`.

## 15a.3. Formula

```text
Formula <Name>
<expression body — một hoặc nhiều dòng, không có block heading khác>
```

Blocks bắt buộc: `Formula`, body.
Không có block optional.

## 15a.4. Rule

```text
Rule <Name>
Kind: Guard | Transformation | Trigger
Trigger:
<mô tả trigger>
Precondition:
<Formula Boolean, optional cho Trigger>
Effect:
<danh sách mutation, cấm với Guard>
Priority:
<Integer, optional, mặc định 0>
```

Blocks bắt buộc: `Rule`, `Kind`, `Trigger`.
Bắt buộc theo Kind:

* Guard          → `Precondition` (không có `Effect`)
* Transformation → `Precondition`, `Effect`
* Trigger        → `Effect`  (Precondition optional)

Optional: `Priority`.

## 15a.5. Event

```text
Event <Name>
Phase: Pre | Post
Trigger:
<mô tả trigger>
Effect:
<danh sách mutation>
```

Blocks bắt buộc: `Event`, `Phase`, `Trigger`, `Effect`.
Không có Precondition riêng — Trigger giữ điều kiện.

## 15a.6. Action

```text
Action <Name>
Intent:
<danh sách phrase>
Parameters:
<name>: <type>
Preconditions:
<Formula Boolean>
```

Blocks bắt buộc: `Action`, `Intent`.
Optional: `Parameters`, `Preconditions`.

Xem PRD-012 §4 để biết cú pháp đầy đủ.

## 15a.7. Query

```text
Query <Name>
<expression body>
```

Blocks bắt buộc: `Query`, body.

Xem PRD-013 §11.

## 15a.8. Nguyên tắc chung

Blocks luôn theo format:

```text
<BlockName>:
<content>
```

BlockName kết thúc bằng dấu hai chấm. Content theo sau, có thể nhiều
dòng, dừng khi gặp BlockName kế tiếp hoặc empty-line-then-EOF.

Chỉ tên block đã liệt kê trong §15a mới được machine-parsed. Bất kỳ
tên block khác → Builder emit `SECTION_SCHEMA_UNKNOWN_BLOCK`.

## 15a.9. Docs vs Rules Split

File markdown chia làm hai lớp:

* **Build-included** — file `.md` được `@import` từ entry point. Chỉ
  chứa Section Schema declaration.
* **Docs-only** — README, gaps.md, notes.md, design/*.md. Không được
  `@import`. Builder bỏ qua.

Comment `<!-- ... -->` được phép trong Build-included file nhưng chỉ
để chú thích. Không có ngữ nghĩa. Được strip khi Bundle.

## 15a.10. Validation Errors

Builder emit các code sau:

```text
SECTION_SCHEMA_MISSING_BLOCK   — thiếu block bắt buộc
SECTION_SCHEMA_FORBIDDEN_BLOCK — block không được phép với Kind (Guard + Effect)
SECTION_SCHEMA_UNKNOWN_BLOCK   — block name không có trong §15a
SECTION_SCHEMA_DUPLICATE_BLOCK — block name lặp trong cùng declaration
SECTION_SCHEMA_MISSING_KIND    — Rule/Entity thiếu Kind block
```

## 15a.11. Documentation Blocks (v1.2)

<!-- Added v1.2. Nguồn: bản Lemonade v2 đã hợp lệ với §15a nhưng thiếu
     ngữ nghĩa reasoning — Rule/Action/Event không nói "để làm gì".
     PRD-014 mở rộng §15a với 4 block tài liệu. -->

Bên cạnh execution blocks (§15a.1-.7), mỗi declaration có thể mang
**documentation blocks**:

```text
Purpose      — vì sao declaration tồn tại (1–3 câu)
Failure      — điều gì xảy ra khi Precondition không hợp lệ
Notes        — bối cảnh / caveat / pointer
Examples     — ví dụ input → output
```

Mức yêu cầu (xem PRD-014 §4):

* Rule / Action / Event: `Purpose` là ERROR.
* Rule Transformation / Guard / Action: `Failure` là WARNING.
* Entity / Variable / Formula / Query: `Purpose` là WARNING.
* `Notes` và `Examples`: luôn optional.

Documentation blocks không thay đổi execution semantics. Runtime bỏ
qua tài liệu này khi tính State (PRD-011 §4). LLM đọc để reasoning.

Xem PRD-014 để biết cú pháp đầy đủ và validator behavior.

# 16. Compatibility

Game Package phải khai báo Runtime Version mục tiêu.

Ví dụ:

```text
Runtime: 1.x
```

Builder từ chối Build nếu Package không tương thích với Runtime đang sử dụng.

---

# 17. Package Independence

Game Package không được phụ thuộc vào Game Package khác.

Một Package phải có khả năng hoạt động độc lập.

Việc tái sử dụng nội dung sẽ được giải quyết bởi Plugin System trong các PRD sau.

---

# 18. Extensibility

Game Package có thể chia thành nhiều module.

Ví dụ:

```text
World

Economy

Combat

Story

Research

Crafting
```

Builder sẽ hợp nhất chúng thành một Game Package duy nhất trước khi Bundle.

---

# 19. Non-goals

Game Package Specification không định nghĩa:

* Runtime
* Compiler
* Directive
* Plugin API
* Save Format
* Multiplayer
* AI Behavior ngoài phạm vi Rule của game

Những nội dung này thuộc các PRD khác.

---

# 20. Tiêu chí hoàn thành

Game Package Specification được xem là hoàn thành khi:

* Một game có thể được mô tả hoàn toàn bằng Markdown theo chuẩn này.
* Runtime có thể chạy nhiều Game Package khác nhau mà không cần sửa Runtime.
* Builder có thể kiểm tra tính hợp lệ của Game Package.
* UI, State và Turn Lifecycle vẫn giữ nguyên, chỉ nội dung game thay đổi.
* Section Schema (§15a) được Builder enforce trên mọi declaration.

reasoning. PRD-014 Documentation Schema mở rộng §15a.

  Failure, Notes, Examples) vào catalog block hợp lệ.

* Failure là WARNING trên Rule Transformation / Guard / Action.
