# PRD-009 — Formula System

**Project:** Markdown Game Runtime (MGR)

**Version:** 1.1

**Status:** Draft

---

# 1. Mục tiêu

Formula System định nghĩa notation để mô tả biểu thức số học và logic
trong MGR.

Formula là lớp nguyên thủy.

Rule dùng Formula để mô tả điều kiện và hiệu ứng.

Event dùng Formula để mô tả xác suất và hệ số.

Không có Formula, Rule và Event chỉ là văn bản. Runtime phải "đoán" — vi
phạm PRD-004 §4 Determinism.

---

# 2. Triết lý

Formula phải:

* deterministic
* readable
* Markdown-native
* độc lập với gameplay
* không phải code

Formula không phải:

* JSON
* JavaScript
* Lua
* biểu thức nhúng trong dấu ngoặc

Formula là cách viết công thức bằng chính Markdown.

---

# 3. Kiến trúc

Formula là một cây biểu thức.

```text
Formula

├── Value       (Integer, Rational, Boolean)
├── Reference   (tên biến State)
├── Operator    (+, -, ×, ÷, mod, =, ≠, <, >, ≤, ≥, AND, OR, NOT)
├── Piecewise   (When ... Then ...)
├── Bound       (Min, Max, Clamp)
├── Aggregation (Sum, Count, Average trên Collection)
├── Random      (Uniform, Weighted)
└── Named       (formula đã đặt tên, tái sử dụng)
```

Builder parse thành cây.

Runtime đánh giá cây.

---

# 4. Value Types

Formula chỉ hỗ trợ ba kiểu giá trị.

```text
Integer     — số nguyên
Rational    — phân số hai số nguyên, ví dụ 3/4
Boolean     — true / false
```

Không có Float.

Float không deterministic giữa các LLM.

Rational giữ được độ chính xác và có thể chuyển sang Integer bằng Round.

---

# 5. References

Formula tham chiếu State bằng tên biến.

```text
Money
Reputation
Lemons
```

Reference resolve tại Build time.

Nếu tên không tồn tại trong State Schema → Build fail.

Runtime không được sinh Reference mới.

---

# 6. Arithmetic

```text
+   cộng
-   trừ
×   nhân   (cũng chấp nhận *)
÷   chia   (cũng chấp nhận /)
mod số dư
```

Ưu tiên toán học chuẩn.

Dấu ngoặc `( )` bắt buộc khi cần rõ ràng.

Ví dụ:

```text
Money - Quantity × 5
```

Là hợp lệ. Nhân trước, trừ sau.

---

# 7. Comparison

```text
=    bằng
≠    khác   (cũng chấp nhận !=)
<    nhỏ hơn
>    lớn hơn
≤    nhỏ hơn hoặc bằng   (cũng chấp nhận <=)
≥    lớn hơn hoặc bằng   (cũng chấp nhận >=)
```

Kết quả là Boolean.

Ví dụ:

```text
Money ≥ Quantity × 5
```

---

# 8. Boolean

```text
AND
OR
NOT
```

Chỉ chấp nhận dạng viết hoa.

Ví dụ:

```text
Money ≥ 100 AND Reputation > 50
```

---

# 9. Piecewise

Formula có thể mô tả nhiều nhánh.

```text
When Weather = Rainy      Then 0.5
When Weather = Heat Wave  Then 2.0
Otherwise                 Then 1.0
```

Nhánh đầu tiên khớp thắng.

Otherwise bắt buộc nếu có nhánh When.

---

# 10. Bounds

Formula phải có notation cho giới hạn.

```text
Min(A, B)
Max(A, B)
Clamp(X, Low, High)
```

Ví dụ:

```text
Clamp(Reputation + 5, 0, 100)
```

---

# 11. Monotonic hints

Cho phép mô tả quan hệ đơn điệu mà không cần công thức chính xác.

```text
Monotonic(Customers, Increases With Reputation)
Monotonic(Customers, Decreases With Price)
```

Runtime coi đây là gợi ý và phải giữ tính đơn điệu trong suy diễn.

Monotonic không thay thế công thức. Nó dùng khi công thức thực sự phụ
thuộc LLM (ví dụ demand curve chưa được tinh chỉnh).

---

# 12. Aggregation

Formula có thể tổng hợp trên Collection.

```text
Sum(Employees.Salary)
Count(Projects)
Average(Reviews.Score)
```

Runtime phải trả kết quả deterministic.

Order-independent (Sum, Count, Average không phụ thuộc thứ tự phần tử).

---

# 13. Randomness

Formula có thể chứa ngẫu nhiên.

```text
Uniform(1, 6)
Weighted(Rainy: 30, Sunny: 60, Heat Wave: 10)
```

Runtime dùng Turn Number làm seed.

Cùng State + cùng Turn + cùng Formula → cùng kết quả.

Điều này đảm bảo Idempotency của PRD-005 §13.

---

# 14. Named formulas

Formula có thể đặt tên và tái sử dụng.

```text
Formula BaseDemand
Reputation × 0.5 + 10
```

Sau đó:

```text
Customers
BaseDemand × WeatherModifier
```

Builder phải resolve tên và cấm định nghĩa lặp.

---

# 15. Reference Resolution

Tại Build time:

* Mọi Reference phải resolve về State Variable, Entity Property, hoặc
  Named Formula.
* Tham chiếu chưa khai báo → Build fail.
* Tham chiếu vòng (A dùng B, B dùng A) → Build fail.

---

# 16. Evaluation Semantics

Runtime đánh giá Formula theo thứ tự sau.

```text
Reference

↓

Value

↓

Operator (arithmetic → comparison → boolean)

↓

Piecewise

↓

Bounds

↓

Aggregation

↓

Random
```

Không side effect. Không mutate State.

Formula là biểu thức thuần túy.

---

# 17. Rounding

Rational chuyển sang Integer phải rõ ràng.

```text
Round(X)
Floor(X)
Ceil(X)
```

Không được ép ngầm.

Ví dụ:

```text
Round(Money × 0.15)
```

---

# 18. Domain & Type Checking

Builder kiểm tra:

* Toán tử số học chỉ nhận Integer hoặc Rational.
* So sánh hai vế cùng miền.
* Boolean operator chỉ nhận Boolean.
* Piecewise When phải trả Boolean.

Sai kiểu → Build fail.

# 18a. Body Structure Enforcement (v1.1)

     xuôi. Đóng khoảng cách này bằng cách enforce body chỉ được là

Formula body PHẢI là một trong các dạng sau:

1. **Single-line expression** — một biểu thức đơn dòng theo §6-§14.
2. **Piecewise block** — một chuỗi `When ... Then ...` kết thúc bằng
   `Otherwise ... Then ...` (§9).
3. **Named Formula reference** — tên một Named Formula khác (§14).
4. **Bounded expression** — `Round(...)`, `Floor(...)`, `Ceil(...)`,
   `Clamp(...)`, `Min(...)`, `Max(...)` bao ngoài expression (§10, §17).

Không được:

* Đoạn văn xuôi tiếng Anh/Việt/... mô tả công thức.
* Nhiều expression trên nhiều dòng không phải Piecewise.
* Comment inline giữa expression (chỉ được HTML comment ở đầu/cuối).
* Directive `@` trong body.

Validator emit `FORMULA_BODY_INVALID` khi body không parse.

Ví dụ sai:

```text
Formula CustomersToday
Customers depend on Weather and Price. When it rains, halve the
demand. Otherwise scale by 1.
```

Ví dụ đúng:

```text
Formula CustomersToday
BaseDemand × WeatherModifier - Price × 0.05
```

Hoặc:

```text
Formula WeatherModifier
When Weather = Rainy      Then 0.5
When Weather = Heat Wave  Then 2.0
Otherwise                 Then 1.0
```

# 19. Non-goals

Formula System không định nghĩa:

* Rule structure     (PRD-010)
* Event lifecycle    (PRD-005)
* State schema       (PRD-006)
* UI                 (PRD-007)
* Loops
* Function definition kiểu ngôn ngữ lập trình
* Recursion
* I/O

---

# 20. Tiêu chí hoàn thành

Formula System được xem là hoàn thành khi:

* Mọi công thức số học và logic của game có thể biểu diễn bằng notation
  này.
* Không cần dùng JSON, JavaScript, Lua trong Game Package.
* Reference resolve deterministic tại Build time.
* Runtime đánh giá Formula cho kết quả giống nhau với cùng State và
  cùng Turn.
* Formula tương thích với PRD-004 §4 Determinism và PRD-005 §13
  Idempotency.
* Body Structure Enforcement (§18a) chặn Formula viết bằng văn xuôi.
