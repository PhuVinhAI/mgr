# PRD-012 — Action Resolution

**Project:** Markdown Game Runtime (MGR)

**Version:** 1.0

**Status:** Draft

---

# 1. Mục tiêu

Action Resolution định nghĩa cách Runtime **chuyển player input** thành
Action có cấu trúc + tham số.

PRD-008 §10 đã nói:

> Người chơi có thể nhập tự do, nhưng Runtime phải ánh xạ vào Action hợp
> lệ hoặc từ chối.

PRD-008 §10 chưa nói:

* làm sao xác định Action player muốn
* làm sao extract tham số từ văn bản
* làm gì khi ambiguous
* làm gì khi input không thuộc Action nào
* reserved intents (Quit, Help, Retry, …)

Action Resolution lấp phần đó.

Nếu không có PRD này, cùng câu "buy 20 lemons" có thể ra Action khác
nhau giữa các LLM — vi phạm PRD-004 §4.

---

# 2. Triết lý

Action Resolution phải:

* deterministic
* explicit (Game Package khai báo Action + parameter shape)
* refuse-by-default (không tự phát minh Action)
* transparent (mọi resolution ghi vào Turn History)
* free-text friendly (cho phép player nhập tự nhiên nhưng không đánh cắp
  ý)

Không phải:

* NLU tự do
* AI-generated action
* prompt engineering trick

Action Resolution là một hàm ánh xạ có cấu trúc.

---

# 3. Kiến trúc

Action Resolution nằm trong Input Phase (PRD-005 §5).

Bốn bước.

```text
Player Input
    │
    ▼
Normalize      — lowercase, trim, unicode fold
    │
    ▼
Match Intent   — chọn Action ứng viên
    │
    ▼
Extract Params — trích tham số từ input
    │
    ▼
Resolved Action
    │
    ▼
→ Validation Phase (PRD-005 §5)
```

Output là một Action object có `name` và `params` đủ để Rule Guard
đánh giá.

Nếu bất kỳ bước nào fail → Runtime trả về `INPUT_UNRESOLVED` và State
không đổi (PRD-005 §5).

---

# 4. Action Declaration

Game Package khai báo Action với cấu trúc cố định.

```text
Action <Name>

Intent:
<danh sách cụm từ khớp>

Parameters:
<name>: <type>

Preconditions:
<Formula trả Boolean, tham chiếu Parameters>

Effect:
<xem PRD-010 §8 hoặc để Rule ApplyX xử lý>
```

Ví dụ:

```text
Action Buy Lemons

Intent:
buy lemons
purchase lemons
grab lemons

Parameters:
Quantity: Integer

Preconditions:
Quantity ≥ 1 AND Money ≥ Quantity × 5
```

`Intent` là danh sách cụm từ chuẩn hóa. Runtime chỉ được match Action
qua Intent phrases đã khai báo.

---

# 5. Intent Phrases

Intent phrases phải:

* viết bằng chữ thường
* không dùng dấu câu
* không dùng số cụ thể (dùng Parameter thay)
* mỗi phrase một dòng

Ví dụ hợp lệ:

```text
Intent:
buy lemons
get lemons
i want lemons
```

Không hợp lệ:

```text
Intent:
Buy 20 lemons     # có số cụ thể — dùng Parameter Quantity
```

Runtime match bằng cách:

1. Normalize input.
2. Với mỗi Action, tính có phrase nào là substring của input không.
3. Nếu có nhiều Action khớp → §7 disambiguation.

---

# 6. Parameter Types

MGR hỗ trợ các kiểu tham số cố định:

```text
Integer     — số nguyên trong input
Rational    — phân số hoặc thập phân → Rational (PRD-009 §4)
Boolean     — yes/no, true/false, on/off
Enum(...)   — một trong các giá trị cho trước
String      — chuỗi tự do có giới hạn
```

Ví dụ:

```text
Action Set Price

Parameters:
Price: Integer

Action Announce
Parameters:
Message: String(max: 80)

Action Choose Ending
Parameters:
Choice: Enum(A, B, C)
```

Không có Float. Không có Array. Không có Object.

Nếu Game cần collection → dùng nhiều Action hoặc Aggregation Formula.

---

# 7. Extraction

Cho mỗi Parameter, Runtime extract giá trị bằng heuristic khai báo trong
Game Package.

Mặc định cho Integer:

* Lấy số nguyên đầu tiên trong input.
* Nếu không có → INPUT_UNRESOLVED.

Mặc định cho Enum:

* Match từng giá trị Enum như một Intent phrase riêng.

Author có thể override.

```text
Action Buy Lemons

Parameters:
Quantity: Integer
  Default: 1
  Match: <regex hoặc phrase, tùy Builder>
```

`Default` được dùng khi input chỉ có Intent mà không có số ("buy lemons"
→ Quantity = 1).

Runtime không được đoán giá trị. Nếu không có Default và không extract
được → INPUT_UNRESOLVED.

---

# 8. Ambiguity Handling

Có ba loại ambiguity.

## Multiple Actions match

Nhiều Action có Intent khớp input.

Ví dụ:

```text
Player: "buy"
```

Match: Buy Lemons, Buy Sugar, Buy Ice.

Runtime **KHÔNG được đoán**. Phải:

1. Emit `INPUT_AMBIGUOUS`.
2. Response phase list các Action có thể.
3. Không mutate State.
4. Chờ input tiếp theo.

## Missing required parameter

Intent match nhưng thiếu Parameter bắt buộc.

Ví dụ:

```text
Player: "set price"
Action Set Price expects: Price
```

Runtime:

1. Emit `PARAMETER_MISSING(Price)`.
2. Response asks for the missing parameter.
3. Không mutate State.

## Type mismatch

Extract được nhưng sai kiểu.

Ví dụ:

```text
Player: "buy negative five lemons"
```

Không extract ra Integer dương → INPUT_UNRESOLVED.

Runtime giải thích lỗi, chờ input mới.

---

# 9. Reserved Intents

Runtime luôn hiểu một số intent bất kể Game Package.

```text
help    — hiển thị Actions available
quit    — xin xác nhận, sau đó end game
retry   — lặp lại last valid Action
undo    — không hỗ trợ (PRD-004 §4 Determinism)
```

Đây là **Reserved Intents**. Game Package không được override.

Runtime phải phân biệt Reserved Intent với Game Action. Nếu input match
cả hai → Reserved Intent thắng.

---

# 10. Input Bypass

Trong một số Phase, Runtime tự tạo Input (không có Player).

```text
Auto-Actions      — Runtime queue Action khi state thoả điều kiện
System-Actions    — End Turn, Save, Load
Event-Actions     — Event fire ra Action (PRD-005 §7/§7a)
```

Auto-, System-, Event-Actions không đi qua Intent Match. Chúng nhập
trực tiếp vào Validation Phase như Action đã resolve.

Game Package khai báo Auto-Action bằng cú pháp:

```text
Auto Action <Name>
Fires When:
<Formula trả Boolean>
```

Runtime evaluate `Fires When` sau khi Player Input được resolve. Nếu
true → enqueue Auto-Action.

---

# 11. Free-text Passthrough

Đôi khi Action cần thu chuỗi tự do (chat, name, message).

```text
Action Say

Parameters:
Text: String(max: 200)
Passthrough: true
```

`Passthrough: true` báo cho Runtime lấy toàn bộ input (sau khi đã match
Intent) làm Text. Không cần Extract riêng.

Cấm nhiều Passthrough Parameter trong cùng Action.

---

# 12. Determinism

Cùng input string + cùng Action set → cùng Resolved Action hoặc cùng
error code.

Runtime không được dùng ngẫu nhiên trong Intent Match.

Không được dùng LLM sampling để đoán Action.

Match theo thứ tự khai báo trong Game Package. Action đầu tiên có Intent
khớp thắng — trừ khi §8 Ambiguity kick in.

---

# 13. Resolution Order

Với mỗi input:

1. Normalize.
2. Match Reserved Intent (§9). Nếu match → hoàn tất.
3. Match Game Action Intents theo Package Order.
4. Nếu 0 match → INPUT_UNRESOLVED.
5. Nếu ≥ 2 match → INPUT_AMBIGUOUS (§8).
6. Nếu 1 match → Extract Parameters (§7).
7. Trả về Resolved Action.

Runtime **không** đi tiếp bước 2 nếu bước 3 trả về ≥ 1 result.

---

# 14. Errors

Action Resolution phát hiện các lỗi sau:

```text
INPUT_UNRESOLVED       — không match Action nào
INPUT_AMBIGUOUS        — match ≥ 2 Action
PARAMETER_MISSING      — thiếu Parameter bắt buộc
PARAMETER_TYPE         — sai kiểu
PARAMETER_OUT_OF_RANGE — vượt Domain (nếu declared)
RESERVED_INTENT_MATCH  — thông tin cho debug, không phải lỗi
```

Runtime chuyển lỗi thành narrative bằng ngôn ngữ tự nhiên trong Response
Phase (PRD-005 §14). State không đổi.

---

# 15. Turn History

Mọi Resolution đều được ghi:

```text
Turn N
  Input: "buy 20 lemons"
  Normalized: "buy 20 lemons"
  Matched: Buy Lemons
  Params: Quantity = 20
  Errors: none
```

Cho phép replay và audit.

Nếu INPUT_AMBIGUOUS, ghi danh sách candidate.

---

# 16. Validation

Builder kiểm tra:

* Mỗi Action có Intent block không rỗng.
* Không có Intent phrase trùng nhau giữa hai Action (`Buy Lemons` và
  `Buy Sugar` không được có phrase chung như `buy`).
* Parameters có kiểu hợp lệ.
* Reserved Intent phrases (§9) không xuất hiện trong Game Intent.
* Auto Action `Fires When` chỉ đọc, không mutate.

Sai bất kỳ → Build fail.

---

# 17. Interaction với Rule Language

Sau khi Resolution xong, Action name và Params vào Guard Rule (PRD-010
§12) trong Validation Phase.

Guard đọc `Parameters` như Reference trong Formula (PRD-009 §5).

Nếu Guard fail Precondition → Action bị reject, Runtime giải thích và
chờ input mới.

---

# 18. Multi-turn Actions

Một số Action cần nhiều lượt tương tác (chọn item → chọn số lượng →
xác nhận).

MGR khuyến khích tách thành nhiều Action nhỏ.

Nếu bắt buộc, dùng State `Pending Action` (declared bởi Game Package)
và Rule để tiếp tục qua các Turn.

MGR không hỗ trợ modal/dialog. Runtime chỉ có Turn Loop.

---

# 19. Non-goals

Action Resolution không định nghĩa:

* Rule execution      (PRD-011)
* Formula             (PRD-009)
* Turn phases         (PRD-005)
* UI                  (PRD-007)
* Save/Load
* Multiplayer input
* Chatbot / small talk
* Voice input

Runtime chỉ nhận text input.

---

# 20. Tiêu chí hoàn thành

Action Resolution được xem là hoàn thành khi:

* Cùng input + cùng Action set → cùng Resolved Action (hoặc cùng error).
* Không có Action match qua Intent chưa khai báo.
* Reserved Intent luôn thắng khi conflict với Game Action.
* Ambiguity không tự resolve — luôn chuyển sang Player.
* Turn History đủ để replay Resolution.
* Interoperable với PRD-010 Guard và PRD-011 Execution.
